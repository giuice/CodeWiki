## Phase Boundary

Phase 4 shipped Claude integration to `.claude/commands/codewiki/`, but the v2 canon requires eight standalone Skills, one per logical command. Phase 04.1 corrects that shipped surface and aligns installer output, tests, and planning docs to the verified skills model.

**CRITICAL — Phase Atomicity Rule:** This phase MUST be broken into sub-phases small enough that any single sub-phase fits within one agent context window (~120K tokens including all read files, plan content, and tool outputs). If a plan touches more than 4-5 files or has more than 3 tasks, it is too large. Split further. See `CLAUDE.md` and `copilot-instructions.md` for the full atomicity constraints.

</domain>

<decisions>
## Implementation Decisions

### Skills canon
- **D-01:** CodeWiki installs eight standalone Skills, not slash commands and not one bundled wiki skill.
- **D-02:** Claude Code uses `.claude/skills/codewiki-<name>/SKILL.md`; Codex, Copilot, and OpenCode use `.agents/skills/codewiki-<name>/SKILL.md`; combined selections use both trees.
- **D-03 (REVISED):** The globally installed `/create-skill` is a **reference only** for skill creation best practices (structure, frontmatter, description quality, discoverability patterns). The eight CodeWiki command files are already well-structured; use `/create-skill` patterns to improve them where relevant, but do NOT use it as the generation mechanism. The skill files are hand-authored/migrated from the existing commands, not generated by `/create-skill`.

### Scope boundary
- **D-04:** Hook scripts and agent definitions remain shared assets; this phase changes the skill install surface and its code/doc/test fallout, not the hook-script feature set.
- **D-05:** `docs/skills/wiki.md` remains a format reference only; CodeWiki does not bundle subcommands into a single skill.

### Phase atomicity (mandatory)
- **D-06:** Every phase and sub-phase MUST be completable within a single agent context window. Maximum 3 tasks per plan, maximum 4-5 files touched per plan. If a plan grows beyond this, split it into a new decimal sub-phase immediately.
- **D-07:** Plans must NOT include doc cascades alongside code changes. Code migration and doc updates are separate sub-phases.
- **D-08:** Each sub-phase must have a single, clear deliverable that can be verified independently.

</decisions>

<specifics>
## Planning Notes

- `/create-skill` is a **quality reference** — consult its patterns for frontmatter, descriptions, and skill structure, but do not use it as a generator. Existing commands are well-structured; migrate them to skill format directly, improving where the reference reveals gaps.
- The migration surface starts with every `commands/codewiki` reference in code, tests, docs, and planning artifacts.
- The expected template source of truth is `src/templates/skills/codewiki-<name>/SKILL.md`, with adapters deciding where to copy those skill directories at install time.

### Suggested Sub-Phase Breakdown (enforce atomicity)

The following is the recommended decomposition — each sub-phase fits in one context window:

1. **04.1a — Template file move**: Move `src/templates/claude/commands/codewiki/*.md` → `src/templates/skills/codewiki-<name>/SKILL.md`. Add frontmatter improvements from `/create-skill` patterns. No code changes, no test changes.
2. **04.1b — Adapter code migration**: Update Claude adapter to write skills to `.claude/skills/codewiki-<name>/SKILL.md` instead of `.claude/commands/codewiki/`. Update dual-tree logic for `.agents/skills/`.
3. **04.1c — Test updates**: Update existing tests that reference `commands/codewiki` paths to use `skills/codewiki-<name>/SKILL.md` paths.
4. **04.1d — Doc cascade (planning docs)**: Update REQUIREMENTS.md, ROADMAP.md, STATE.md references from commands to skills.
5. **04.1e — Doc cascade (project docs)**: Update README.md, implementation-plan-v2.md, research docs.

Each sub-phase: ≤3 tasks, ≤5 files, independently verifiable.

</specifics>
