Create `src/templates/claude/commands/codewiki/absorb.md` with the following structure.

The file must have YAML frontmatter (between triple-dash delimiters) with: `description: Extract knowledge from recent code changes into wiki` and `allowed-tools: [Read, Write, Edit, Glob, Grep, Bash]`.

After frontmatter, use the same XML structure as ingest.md (`# CodeWiki Absorb` heading, then `purpose` and `process` XML tags).

Purpose section: Read recent git changes, cross-reference against the wiki, and propose updates that capture new knowledge. This is code-focused absorption — read git diff and changed files, not raw documents.

Process steps (each as `## Step N: Title` inside the process tag):

Step 1 Gather recent changes: Run `git diff HEAD~1` (or `git diff --cached` if uncommitted). List changed files and summarize what changed semantically (not line-by-line).

Step 2 Load wiki state: Read `wiki/index.md`. Read `wiki/_backlinks.json`. Glob `wiki/**/*.md` to get full inventory.

Step 3 Cross-reference changes against wiki: For each changed file, search wiki for existing pages that reference it. Identify: (a) existing pages that need updating, (b) new entities/decisions/lessons implied by the changes, (c) contradictions between changes and current wiki content.

Step 4 Apply anti-cramming rule: If any existing page would gain a 3rd paragraph about a sub-topic, propose creating a new dedicated page instead. Prevent bloated articles — prefer 30 focused pages over 5 fat ones.

Step 5 Apply anti-thinning rule: Every page touched must get meaningfully richer. If a topic is mentioned in 4+ other pages, its own page must not be a 3-sentence stub. Flag thin pages for expansion.

Step 6 Update backlinks: After proposing changes, rebuild `wiki/_backlinks.json` by scanning all wiki pages for `[[wikilink]]` references. Structure: `{ "page-path": ["referencing-page-1", "referencing-page-2"] }`.

Step 7 Propose changes: Present a summary: pages to create, pages to update, backlinks changes, contradictions found. Wait for user approval before writing.

Step 8 Write approved updates: After approval, create/update only approved pages using `.codewiki/templates/`. Update `wiki/index.md` and `wiki/_backlinks.json`.

Step 9 Guardrails: Never write wiki without approval. Never auto-commit. If changes are trivial (only whitespace/formatting), say so and skip.

Create `src/templates/claude/commands/codewiki/breakdown.md` with the following structure.

YAML frontmatter with: `description: Find undocumented entities referenced in wiki and propose new pages` and `allowed-tools: [Read, Glob, Grep, Bash]`.

After frontmatter, use the same XML structure as lint.md (`# CodeWiki Breakdown` heading, then `purpose` and `process` XML tags).

Purpose section: Scan the wiki for concrete entities (functions, modules, patterns, decisions) that are mentioned but have no dedicated page. Rank candidates by backlink count and propose new pages in batches.

Process steps:

Step 1 Load backlink index: Read `wiki/_backlinks.json`. If empty or missing, scan all `wiki/**/*.md` files for `[[wikilink]]` references and build the index first.

Step 2 Load wiki inventory: Read `wiki/index.md`. Glob `wiki/**/*.md` to get all existing pages.

Step 3 Find undocumented references: For each `[[wikilink]]` target in the backlink index, check if a corresponding wiki page exists. Collect all targets that have no page.

Step 4 Scan code for additional candidates: Grep the codebase (src/, config files) for entity names mentioned in wiki pages but not documented. Look for: exported functions, class names, config keys, CLI flags, environment variables.

Step 5 Rank by importance: Sort candidates by: (a) backlink count from `_backlinks.json` (higher = more important), (b) number of wiki pages that mention the entity, (c) whether the entity appears in code (code-backed entities are more valuable).

Step 6 Propose batch: Present the top 5-10 candidates with: entity name, reference count, suggested wiki category (entity/decision/lesson/issue), one-line description. Wait for user to select which to create.

Step 7 Create approved pages: For each approved candidate, create a wiki page using `.codewiki/templates/`. Update `wiki/index.md` with the new entries. Update `wiki/_backlinks.json`.

Step 8 Guardrails: Never create pages without approval. Batch size max 10 per run to avoid context overload. If no gaps found, say so explicitly.
