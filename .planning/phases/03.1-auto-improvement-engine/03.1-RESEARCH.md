# Phase 3.1 Research: Auto-Improvement Engine

## Source References

The auto-improvement engine design is drawn from three external references. The planner should read the full source files for deeper context.

### 1. Farzaa's LLM Wiki Skill
**File:** `docs/skills/wiki.md`
**Key patterns to borrow:**
- **Absorption loop** (§`/wiki absorb`): Process entries chronologically, read index before each, understand meaning not just facts, update+create articles, connect to patterns
- **Anti-cramming**: If adding a 3rd paragraph about a sub-topic, create a new page instead. Prevents 5 bloated articles instead of 30 focused ones.
- **Anti-thinning**: Every page touched must get meaningfully richer. A stub with 3 vague sentences when 4 other entries mention that topic is a failure.
- **Checkpoint every 15 entries**: Rebuild index, rebuild backlinks, audit article quality (diary-driven vs narrative), check line counts (>120 = split)
- **Breakdown mining** (§`/wiki breakdown`): Scan wiki for concrete entities mentioned but undocumented, rank by reference count, create in batches
- **Backlink index** (`_backlinks.json`): Reverse link map rebuilt by scanning `[[wikilinks]]`. High backlink count = important topic.
- **Quality audit rules**: Articles must be organized by theme not chronology. Direct quotes carry emotional weight. Every article must have a point.

### 2. Karpathy / Nick Spisak Knowledge Base Pattern
**File:** `docs/how-to-get-second-brain.md`
**Key patterns to borrow:**
- **Output filing**: AI answers get saved back to the knowledge base, not lost. Every question makes the next answer better.
- **Compounding loop**: The wiki gets richer with every interaction, not just manual ingestion.
- **Monthly health check**: Flag contradictions, find topics mentioned but never explained, list unsourced claims, suggest gap-filling articles.
- **Error compounding risk**: When outputs get filed back, errors compound too. Solution: periodic health checks + human approval gate.

### 3. Our Design Adaptations
**What we changed from the references:**
- **Separate commands, not unified skill**: For token efficiency. Each command prompt is loaded independently — the AI tool only reads what it needs. Farzaa uses one SKILL.md with subcommands; we use `/codewiki-absorb`, `/codewiki-breakdown` as separate files.
- **Human approval gate on everything**: Farzaa's skill writes wiki directly. We always propose first, human approves. This prevents error compounding (Spisak's concern).
- **Code-focused, not personal data**: Farzaa's skill is for personal journals/notes. Our absorb reads `git diff` and code changes, not diary entries. Wiki page types are entities, decisions, lessons, issues — not people, places, eras.
- **Session-end hook as auto-trigger**: Neither reference has this. We add a hook that fires on session end to automatically trigger absorb, so knowledge capture happens even when the dev forgets.
- **Post-hook triggers wiki-updater agent**: The post-hook doesn't just remind — it passes change context to the wiki-updater agent which proposes concrete edits.

## Key Design Decisions for the Planner

1. **absorb.md** must instruct the AI to read `git diff`, not raw/ entries. This is a code wiki, not a journal wiki.
2. **breakdown.md** must use `_backlinks.json` for ranking. Don't scan all files on every run — read the pre-computed index.
3. **_backlinks.json** starts as `{}` in the scaffold. Updated by absorb, ingest, and lint. Structure: `{ "page-path": ["referencing-page-1", "referencing-page-2"] }`.
4. **session-end.sh** must be lightweight — just output git diff summary. The AI tool does the reasoning.
5. **post-verify.sh** already exists and works. Update it to output structured context, not just a reminder string.
6. **Existing prompts** (ingest.md, query.md, lint.md) need small updates to reference `_backlinks.json`.
